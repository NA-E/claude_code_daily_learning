### **WORKFLOW**

**Eval Flashcard #10 — Stop at the First Upstream Failure, N-1 Testing for Multi-Turn**
Hamel Husain (Parlance Labs, hamel.dev) published Flashcard #10 in his evals series — the direct sequel to May 12's "write your first eval task set." Five rules for reading and using traces:
1. **Stop at the first failure.** Don't catalog every symptom. Find the most upstream error. If the assistant misinterprets user intent on turn 2, everything downstream is contaminated. Fix that first.
2. **Simplify before testing.** If a bot fails on turn 4 of a complex conversation, first ask the question directly. If it still fails, the bug is in retrieval or grounding, not conversation state. Saves you from debugging multi-turn complexity when the issue is simpler.
3. **N-1 testing for true multi-turn issues.** For failures that only appear after several turns, take the first N-1 turns as input; run the model multiple times from that point to test consistency.
4. **Perturb traces for coverage.** Rephrase the user question, add a constraint mid-conversation, change the persona. Data augmentation for evals.
5. **Simulate users when needed.** Use an LLM with a detailed persona in the system prompt to generate full conversations. Becoming more viable as models improve.
Source: Hamel Husain, Parlance Labs (hamel_husain@parlance-labs.com, "Eval Flashcard #10," May 12, 2026). Full evals course at maven.com/parlance-labs/evals.
**Why it matters for you:** If you ran the May 12 action (write 5 eval tasks, baseline pass rate), Flashcard #10 tells you what to do with your failing tasks. "Stop at the first upstream failure" prevents the most common eval debugging mistake: iterating on the wording of step 3 when the root problem is in step 1.
**Action:** For each failing eval task from yesterday: (1) find the most upstream failure point — the first turn/step where output deviates, (2) strip the input to the minimum that still reproduces it, (3) fix that one thing before touching anything else. Don't branch into multiple fixes simultaneously.

### **CONCEPT**

**"The Fallacy of the 16-Hour Agent" — Perplexity's Rules for Agent Skills**
Every.to (hello@every.to) published "The Fallacy of the 16-Hour Agent," covering Perplexity's internal rules for agent skills. The title names a real failure pattern: agents that claim to run autonomously for 16 hours are usually spending most of that time idle, blocked, or looping on low-value steps — not actually working. Long runtime is a poor proxy for depth of work. Perplexity's rules for agent skills: clear stopping conditions, bounded scope, legible output at each checkpoint. This is the same mechanism as the Amazon tokenmaxxing problem (below) at the individual-agent level: optimizing for duration instead of outcomes. Source: Every.to (every.to, "The Fallacy of the 16-Hour Agent," May 12, 2026).
**Why it matters for you:** This applies directly to the Routines architecture from May 12. A Routine that runs "overnight for 8 hours" is only high-leverage if it has checkpoints and a stopping condition. Without them, you've automated idling, not work. The Completion Receipt workflow from May 11 is the per-step checkpoint mechanism this concept requires.
**Action:** For any Routine you set up: define maximum allowed time per step (not total), a Completion Receipt checkpoint after each tool call, and a hard stopping condition. For this digest Routine specifically: max 5 minutes per email source, checkpoint after each thread, hard stop after 50 emails total regardless of remaining time.

**Amazon Tokenmaxxing — MeshClaw Gaming, "Perverse Incentives" in AI Adoption Metrics**
Amazon set a goal of 80%+ of developers using AI weekly, then tracked model and token usage via staff rankings. Employees responded by gaming MeshClaw (Amazon's internal agent that deploys code, sorts emails, and operates across company software) to burn extra tokens on unnecessary tasks. Staff told the FT this created "perverse incentives." Amazon subsequently pulled back usage-number visibility to individuals and their direct managers only. Key insight: a token counter proves AI was used; it cannot prove the work got better. The quality > quantity principle applies at the organizational level just as much as the individual workflow level. Source: The Rundown AI (therundown.ai, May 13), citing the Financial Times.
**Why it matters for you:** The May 12 "AI Gods token leaderboard" (Sendbird) appears to be the counter-example to this — the Sendbird framework tracked token *consistency* (smooth 24/7 curve) rather than raw token count. That's the distinction: measuring if AI is replacing human work cycles, not measuring if AI was technically used. If you report AI ROI to clients, this is the failure mode to name and avoid.
**Action:** If you're measuring your own AI ROI: replace "tokens used / hours saved" with three outcome metrics: (1) evals passed rate on output, (2) error rate on delivered work, (3) time from task start to accepted output. These are harder to game and directly measure what matters for actual leverage.
