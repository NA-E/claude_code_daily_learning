**CONCEPT**

**Andrew Ng: "AI Tokens Are Cheap, Human Tokens Are Gold" — Spec-Driven 0-to-1 Prototyping**
*Source: The Batch, DeepLearning.AI (thebatch@deeplearning.ai, "Restoration of Claude Fable 5, Gemini's Video Dev Engine, DeepSeek Speeds Up Speculative Decoding")*

Ng's core argument for building quick prototypes: don't spend an hour writing a careful spec when a coding agent can build a rough version in 20 minutes — spend 10 minutes on an inferior spec, let the agent build it, examine what it got wrong (things you couldn't have predicted, like confusing UI choices), then refine and repeat. His organizing principle: since a coding agent may burn millions of tokens per project versus what a human would type, agent tokens are disposable but human review time is the scarce resource — so optimize for fast iteration loops, not upfront correctness. He recommends steering the agent to record key decisions in a `SPEC.md` file so they survive context compaction.

*Why it matters for you*: Directly reframes how to approach the McFarland second-brain build this repo has been recommending since July 10 — don't try to get the skeleton prompt perfect first; build it rough today, see what breaks, then fix the spec.

*Action*: For your next 0-to-1 Claude Code build, skip the careful spec — write a 2-sentence rough spec, let the agent build for 20 minutes, then spend your review time only on what surprised you, not on re-reading the whole output.

**CONCEPT**

**PwC's Prefix Caching Study: 80% Cost Cut, 31% Faster Time-to-First-Token Across 500+ Agentic Sessions**
*Source: The Nuanced Perspective, Substack (thenuancedperspective+the-chai-and-ai-community@substack.com, "Prefix Caching for Builders")*

PwC ran 500+ agentic sessions across OpenAI, Anthropic, and Google and found the best caching strategies cut cost by ~80% and time-to-first-token by 31% — but naively caching the whole context often made latency *worse*. The mechanism: providers match exact tokens left-to-right and stop at the first difference, so a single dynamic value near the top of your prompt (a timestamp, reordered JSON keys in tool definitions) breaks the cache for everything below it. The fix is a strict layout rule: put system instructions, tool definitions, skill bodies, output contracts, and few-shot examples — in that exact order — in a stable prefix, and put anything that changes (timestamps, user input) at the very end.

*Why it matters for you*: This is the concrete mechanism behind every token-routing number this repo has cited since July 8 (Anthropic's 92%/63% advisor pattern, The Rundown's 60% savings claim) — none of those numbers hold if your prompts aren't cache-stable to begin with.

*Action*: Audit one Claude Code system prompt or CLAUDE.md file today — move any timestamp, session ID, or dynamically-ordered tool list to the very end of the prompt, and confirm tool definitions are serialized in a fixed key order.

**WORKFLOW**

**Sabrina Ramonov's 5-Part /goal Prompt Template + 6 Copy-Paste Loop Engineering Projects**
*Source: Sabrina Ramonov, Substack (sabrinaramonov@substack.com, "6 INSANE Projects to Learn Claude Fable and /goal")*

Follow-up to her prior "Loop Engineering" post (referenced in this repo's July 1 entry). Her template: every `/goal` prompt needs TASK (the verb), WHY (context/audience — she flags this as especially important for Fable), OUTCOME (a specific finished result, not "make it good"), CONSTRAINTS (guardrails, turn limits), and VERIFICATION (a measurable check, e.g., "spider reaches the top 8 out of 10 times"). Six example projects: a Genesis-World physics-sim spider that learns to climb stairs (~60 min), a browser-playable dragon-flying game (requires Claude in Chrome for self-testing), an Apify-scraped YouTube-trends dashboard, a week of content, a 24/7 DM A/B test, and a trip itinerary.

*Why it matters for you*: Gives you a concrete, testable prompt structure to pair with Ng's "build rough, iterate fast" principle above — VERIFICATION is the piece most builders skip, and it's the one that actually stops a loop instead of running forever.

*Action*: Before your next `/goal` command, write out the VERIFICATION line first — a specific, checkable pass/fail condition — before writing TASK or OUTCOME.
