### **CONCEPT**

**Context Window as the Unifying Problem in Claude Code**
The Nuanced Perspective published a full first-principles breakdown of Claude Code. The core insight: every Claude Code feature — CLAUDE.md, skills, hooks, subagents, MCP — solves the same problem: *"how to keep the right information in front of the model and the wrong information out."* There are exactly two failure modes. Context overflow: in a long session, older instructions get crowded out as the window fills. Convention drift: nothing you tell Claude survives between sessions, so every new session starts from scratch and preferences you mentioned once are simply gone. Every feature addresses one or both of these.
- **Source:** The Nuanced Perspective / Substack (thenuancedperspective+the-chai-and-ai-community@substack.com) — "Claude Code 101: A First Principles Approach," June 20, 2026
- **Why it matters for Nourin:** This reframes everything in your setup. Before adding any Claude Code feature, ask: "which failure mode does this solve?" If neither, don't add it.
- **Action:** Audit your current CLAUDE.md and note which lines solve context overflow vs convention drift vs neither. Lines that solve neither should be cut.

**Claude Code Feature Ladder: Move Up Only When Forced**
From the same piece: Claude Code features are a *ladder*, not a menu. You climb rungs only when a specific frustration forces you to. Level 1 = raw prompting (the right tool for one-off tasks more often than people admit). Level 2 = CLAUDE.md (solves convention drift — preferences not persisting between sessions). Level 3 = skills + hooks (triggered context — load the right thing at the right moment). Level 4 = subagents (parallel work, separate context windows). Moving to a higher level before you've hit the lower level's limits is overhead, not improvement.
- **Source:** The Nuanced Perspective (thenuancedperspective+the-chai-and-ai-community@substack.com), June 20, 2026
- **Why it matters for Nourin:** Your prior entries show you're already using CLAUDE.md (May 19, June 11, June 17 all reference it). That's Level 2. The question is whether you're ready for Level 3 — and the answer is: only if you're repeating context across sessions that CLAUDE.md alone can't solve.
- **Action:** List 3 recurring tasks where you find yourself re-explaining context every session. If those tasks follow a pattern, they're candidates for a skill (a named, loadable prompt template). If they need to run on every session start, they're candidates for a hook.

**Model Evaluation Overfitting: Patching vs. Proving Improvement**
The Neural Maze published a piece by Mai Nguyen (formerly Zapier AI Platform). Core argument: when you fix LLM failures one at a time, your metrics go up but you may not be improving the system — you may just be memorizing the specific cases you saw. The key question that distinguishes real improvement from overfitting: *"Did this actually get better, or did I just memorize the cases I happened to see?"* Proper fix: before patching any Claude output failure, run it against a holdout set that the patch hasn't seen. If the holdout score doesn't improve, you haven't fixed the model — you've memorized the failure.
- **Source:** The Neural Maze (theneuralmaze@substack.com) — "You Didn't Fix the Model. You Memorized the Failure," June 20, 2026
- **Why it matters for Nourin:** The same failure mode applies to CLAUDE.md tuning. Every time you add a rule to CLAUDE.md to fix a specific bad output, you're doing prompt patching. Without testing against outputs the new rule hasn't seen, you don't know if the fix generalizes.
- **Action:** Next time you add a rule to CLAUDE.md to fix a bad output, immediately run it against 5 other similar prompts that weren't the original failure case. If those 5 still fail, the rule is too narrow.

## Top Thing to Implement Right Now

**ONE ACTION → HIGH LEVERAGE → DO TODAY**

**Write or rewrite your CLAUDE.md using the convention-drift framework from The Nuanced Perspective.**

The single highest-leverage move is to stop the convention drift tax — the repeated cost of re-explaining your preferences at the start of every session. Here's the exact method:

**Step 1 — List your last 5 session openers.** What did you tell Claude in the first 3 messages of your last 5 sessions? Every phrase you repeated is a convention that belongs in CLAUDE.md.

**Step 2 — Categorize each by failure mode:**
```
Convention drift → goes in CLAUDE.md (survives between sessions)
Context overflow → goes in a skill (loads only when needed, doesn't bloat every session)
Neither → delete it
```

**Step 3 — Write one line per convention.** No paragraphs. CLAUDE.md is a ruleset, not a README. Example:
```markdown
# My Development Conventions
- Default language: Python 3.11
- Prefer explicit over implicit: no magic imports, no hidden side effects  
- Always write tests in the same file as the function being tested (pytest)
- When uncertain about intent, ask before acting — don't guess
- Code comments only where the WHY is non-obvious; never explain WHAT
```

**Step 4 — Add one fallback model line at the top:**
```markdown
model: claude-opus-4-8
```

This takes 20 minutes. It will save you 5-10 minutes of re-explanation in every future session, compounding indefinitely. The June 17 entry told you to audit Fable 5 references — this is the natural next step: having a clean, intentional CLAUDE.md that survives the next model disruption.
