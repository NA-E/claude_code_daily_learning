**Claude Managed Agents: Traces — See Exactly Where Your Agent Got Stuck**
From the same Jess Yan episode (timestamp 09:28): Traces are Anthropic's built-in observability layer for long-running Claude agents. Every step the agent takes — tool calls, reasoning branches, sub-tasks — is logged as a trace. When an agent fails or stalls, you open the trace and see the last successful step, what the agent attempted next, and what caused the halt. This is the production debugging workflow Anthropic uses internally, not a prototype. Without traces, a stalled agent is a black box; with traces, every failure becomes a readable error log. Source: Creator Economy / Peter Yang Substack, June 28.
**Action:** If you're using Claude API agents, add `trace=True` to your agent runs (or use Anthropic's agent console). For Claude Code specifically, review the `.claude/transcripts/` directory after any failed multi-step task — this is the manual equivalent of what traces do automatically.

**Evals as Exit Condition — How Anthropic Agents Grade Their Own Work**
Timestamp 15:11 of the Jess Yan episode: Anthropic agents don't stop when they think they're done — they stop when they pass evals. Each agent has a rubric (2–5 success criteria) defined before it starts. This is the same "eval-as-exit-condition" pattern Gumloop shipped as a platform feature (June 27 entry) and Shreya Shankar published in the Maven course (June 26 entry). Three sources, same week, same pattern — it's the production standard. Source: Creator Economy / Peter Yang Substack, June 28.
**Action:** Before your next Claude Code headless run, add 3 explicit success criteria at the end of your prompt: `"You are done only when: [criterion 1], [criterion 2], [criterion 3]. Check each before returning your result."` This single change converts Claude Code from a one-shot tool to a self-correcting loop.

**The 5-Folder Claude Workspace (Your Folder Is Part of Your Prompt)**
Artificial Corner (artificialcorner@substack.com) published the definitive folder-structure guide for Claude, June 28. Core thesis: "A prompt isn't just the text you type. It's also everything in your folder." Messy folder = Claude opens the wrong draft, treats old notes as current, and spends tokens figuring out your workspace. The 5-folder structure:
```
YOUR-WORKSPACE/
├── [01] system/        # Rules Claude must follow
├── [02] context/       # Stable context Claude must know (library, no order)
├── [03] projects/      # Active work (numbered workflow inside)
├── [04] outputs/       # Completed work
└── [99] archive/       # Old but searchable
```
Inside `[03] projects/`, use numbered sub-steps: `00_BRIEF.md`, `01_SOURCES/`, `02_NOTES/`, `03_DRAFTS/`, `04_VISUALS/`, `05_FINAL/`. File naming rule: `YEAR-MONTH-DAY_descriptive-name_status.ext`. The `[02] context/` folder is a library, not a workflow — no numbers, just descriptive names (`HOW-I-WORK.md`, `voice-guide.md`, `audience.md`). Source: Artificial Corner Substack, June 28.
**Action:** This week, restructure the folder Claude touches most often using this schema. Start with one project: rename files in it to `YEAR-MONTH-DAY_name_status.md` format and move old versions to `[99] archive/`. Do not reorganize everything tonight — one folder is enough to see the difference in Claude's output quality.

**CONCEPT**

**File Names as Context — Numbers for Order, Names for Meaning, Dates for Search**
The Artificial Corner guide distills the rule to one sentence: "Markdown for instructions. Numbers for order. Names for meaning. Dates for search. Archive for noise." Concretely: numbered prefixes `[01]`, `[02]`, etc. tell Claude the priority/sequence to read files in; descriptive file names let Claude infer content before opening them; ISO dates (`2026-06-28`) make everything time-searchable; the `[99] archive/` suffix keeps old work out of active context without losing it. Applied to Claude Code: your CLAUDE.md functions as `[01] system/SYSTEM-RULES.md`, and any persistent brand/voice/project context belongs in `[02] context/` — not scattered in the project root. Source: Artificial Corner Substack, June 28.
**Action:** Rename your CLAUDE.md to reflect this system: treat it as your `[01] system/` layer. Create a `[02] context/` folder adjacent to it with at least one reusable file (`HOW-I-WORK.md` or `voice-guide.md`) that Claude can load without re-prompting.

**Build your 3-foundation-file Claude second brain in the next 30 minutes.**

The Mike Futia system + the Artificial Corner folder structure converge on the same architecture. Here's the exact sequence:

```bash
# 1. Create the context folder in your main Claude workspace
mkdir -p ~/claude-workspace/[02]\ context/

# 2. Create the three foundation files (fill in your actual content)
cat > ~/claude-workspace/[02]\ context/brand-dna.md << 'EOF'
# Brand DNA
[Your name / project], [what you do], [who you serve]
Core differentiator: [what makes you different]
Competitors: [who they are, what they do differently]
EOF

cat > ~/claude-workspace/[02]\ context/voice-guide.md << 'EOF'
# Voice Guide
Tone: [direct/warm/technical/casual]
Do: [specific phrases or patterns to use]
Don't: [specific patterns to avoid]
Example of good output: [paste one]
EOF

cat > ~/claude-workspace/[02]\ context/icp.md << 'EOF'
# Ideal Customer Profile
Who: [role, industry, company size]
Pains: [top 3 problems they have]
Desires: [top 3 outcomes they want]
Objections: [top 3 reasons they don't buy]
Language they use: [actual phrases from conversations/reviews]
EOF

# 3. Add this line to your CLAUDE.md
echo '
Before any creative or strategic task, read [02] context/brand-dna.md, [02] context/voice-guide.md, and [02] context/icp.md first.' >> ~/claude-workspace/CLAUDE.md
```

Futia stood this up in 30 minutes of editing. You already have this repo's 19 entries as the proof that building context into Claude's startup state changes output quality — this is the personal-brand version of the same pattern. Do it today; the weekly self-improvement loop can come after.
