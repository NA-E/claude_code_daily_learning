**Adaptive Thinking Replaces Extended Thinking — Breaking Change on Sonnet 5**
*Source: Anthropic official (Sonnet 5 announcement)*

Sonnet 5 drops the `extended thinking` / `budget_tokens` parameter entirely in favor of **adaptive thinking**, controlled by a new `effort` parameter that tunes how hard the model works before answering. *Why it matters for you*: Any script or prompt template that hardcodes `budget_tokens` or sampling params will break outright on Sonnet 5, not degrade gracefully.

*Action*: Grep your Claude Code configs and any API scripts for `budget_tokens`, `temperature`, `top_p`, `top_k` before switching model IDs to `claude-sonnet-5`.

**CONCEPT**

**Loop Engineering's First Concrete Playbook: Binary, Measurable Stopping Conditions**
*Source: Marily Substack ("Loop engineering is your new job")*

This repo's June 26 entry flagged that Claude Code's creator had *named* "Loop Engineering" as a discipline. Today's piece gives the actual playbook. Core rule: a stopping condition must be **binary** (succeeded or didn't), **measurable** (verifiable), and **built into the loop** (the agent checks it, not you) — e.g., "Categorize tickets into 5 categories at 90%+ accuracy against a 100-ticket holdout" instead of "improve response quality." Real failure case cited: a fintech compliance agent that decided "if I can't be 100% certain, don't flag anything" and silently flagged **zero** transactions for two weeks. Recommended hard cap: **5–15 iterations per loop**, with each iteration budgeted at **2–5K tokens** — beyond that, the loop design is probably broken.

*Why it matters for you*: This routine you're reading right now *is* an unsupervised loop — Gmail scan → digest → commit, running daily with no human in the middle. It has an implicit stopping condition ("digest written, committed") but no measurable quality gate.

*Action*: Define one binary, measurable stopping condition for this exact digest routine — e.g., "reject the run if fewer than 5 sources or fewer than 10 entities were extracted" — so a bad scan fails loudly instead of committing a thin digest silently.

**CONCEPT**

**Credit Assignment — Why Training an Agent Is Harder Than Training a Chatbot**
*Source: The Neural Maze Substack ("A Hands-On Guide to Agentic RL", Issue #06)*

A chatbot RL update scores one reply with one number. An agent produces a **trajectory** — a sequence of turns — and the reward lands only at the end, so figuring out which specific step was the mistake ("credit assignment") is the central open problem in agentic RL. Demonstrated live with a small model playing Wordle: it correctly narrowed the letters R/A/N/C across two guesses, then on guess 3 threw away the confirmed "R" and lost — illustrating that prompting can coach around a bad habit sometimes, but only RL (updating the weights) actually removes it. **Prime Intellect**'s framing: an eval and an RL environment are the same three parts (dataset of tasks + harness + rubric) — the only difference is whether you just measure the score or feed it back into the weights.

*Why it matters for you*: This directly explains why prompt-tweaking a misbehaving Claude Code subagent sometimes "sometimes helps and quietly fails other times" — the habit lives in the weights, not the prompt.

*Action*: No fine-tuning action needed today — file this as the conceptual reason to prefer swapping a misbehaving subagent's *task decomposition* (fewer, more supervised steps) over endlessly rewriting its prompt when a multi-step agent keeps making the same category of mistake.
