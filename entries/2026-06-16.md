**Pietro Schirano's Self-Writing /goal Prompt — Agent Defines Its Own Mission Before Working (The Neuron)**
Pietro Schirano (creator of MagicPath) shared a workflow: instead of writing the agent's `/goal` yourself, ask it to write one, plus a goal for each sub-agent it will spawn. He "basically never" writes his own `/goal` anymore. Verbatim prompt to use before any multi-step agent task:
```
Before starting, write your own /goal for this task.

Task: [describe the task]
Context: [paste files, docs, requirements, or links]
Constraints: [scope limits, style rules, deadlines, things to avoid]
Definition of done: [what success looks like]

Return:
1. Your main /goal.
2. 3–5 success criteria.
3. Boundaries you should not cross.
```
Human reviews the /goal and tightens constraints. Only then does the agent begin. Steven Cheng in the replies flagged that spawned agents without human-set boundaries drift into edge cases — the human-in-the-loop review step is what prevents that.
*Source: The Neuron / Pietro Schirano via X (theneuron@newsletter.theneurondaily.com), June 16, 2026*
**Why it matters for you:** The May 16 entry documented Claude Code's `/goal` command shipping for unsupervised loops. This is the practical usage pattern that makes `/goal` actually reliable — asking Claude to write its own mission spec before execution rather than you writing it cold. The May 12 entry's "Routines" pattern needed this step — the model knowing what "done" looks like is what prevents runaway loops.
**Action:** Add this prompt as a reusable snippet in your CLAUDE.md or as a saved Claude.ai Project instruction. The next time you kick off a multi-step Claude Code session, run this first and review the returned /goal before saying "go."

### **CONCEPT**

**Model-Agnostic Doctrine — "Own the Playbook, Rent the Tech" After Fable 5's 3-Day Shutdown (The AI Exchange / AMP, theaiexchange@mail.beehiiv.com)**
The AI Exchange (now "the AMP Team") framed the Fable 5 ban's real lesson: hundreds of teams had workflows that stopped working when Fable 5 was pulled three days after launch — because they'd built around one model's quirks rather than writing the process in a document they own. Their principle: "Own the Playbook, Rent the Tech." Process: write every AI workflow as a plain doc — inputs, steps, what a good result looks like. Feed that doc to any model. When the model changes, you swap the model, not the process. Their test: take one workflow, run it through both Claude and ChatGPT, observe the gap in outputs. "That gap is your to-do list — it shows you exactly where your playbook is leaning on one model to fill in the blanks." Additional signal from The Rundown: Microsoft's Satya Nadella published a memo introducing "token capital" — AI your company owns rather than rents — and a "learning loop" system where removing one model and dropping in another doesn't disrupt institutional knowledge, because that knowledge lives in your system, not the model.
*Source: The AI Exchange / AMP Team (theaiexchange@mail.beehiiv.com) + The Rundown (news@daily.therundown.ai), June 16, 2026*
**Why it matters for you:** Every prior entry in this repo has documented model-specific behaviors, pricing, and benchmarks. This is the conceptual frame that makes that collection durable: if the workflow doc lives here and the model is a variable, no future Fable ban or SDK credit reversal breaks the practice. Nadella's "token capital" maps exactly to what CLAUDE.md is already doing — your instructions are the institutional knowledge, not the model's weights.
**Action:** Pick the single AI workflow you run most frequently. Write it out as a plain-text doc (inputs → steps → what good looks like). Store it in this repo under `workflows/`. Then run the same doc through both Claude and ChatGPT and note where the outputs diverge. That divergence is your next prompt-engineering task.

**Transformer Context Window Degradation — Why Long Chats Fall Apart and What to Do (Artificial Corner, artificialcorner@substack.com)**
Artificial Corner distilled 10+ AI research papers into practical implications. Key finding on the Transformer architecture: the model compares everything in your message with everything else simultaneously (not sequentially). Doubling prompt length nearly quadruples the computational work. When the context window fills, earlier content is forgotten with no warning. The fix: place critical instructions at both the start and end of long prompts. For conversations that have run too long, start fresh with a structured summary of decisions made. "Noise takes up space too" — unnecessary context degrades answer quality, not just length.
*Source: Artificial Corner (artificialcorner@substack.com), June 16, 2026*
**Why it matters for you:** The 2026-06-11 Fable 5 entry documented that the model sustains context across millions of tokens better than Opus — but "better" doesn't mean "infinite." This explains the mechanism: the agent-chains-to-depth-10 finding from Huryn's June 12 entry has a physical cause (quadratic compute cost). For any Claude Code session that's running multi-hour tasks, the right architecture is periodic context summarization checkpoints, not one continuous session.
**Action:** For any Claude Code task that will span more than 30 minutes, add an explicit checkpoint: every 5–7 turns, ask Claude to "summarize the decisions made and the current state before continuing." Save that summary. If the session drops, restart from the summary, not from scratch.
