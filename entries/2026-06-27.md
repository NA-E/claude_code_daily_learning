**Vapi Voice AI Production Framework: Impact × Feasibility Matrix**
Vapi CTO Nikhil Gupta (published in Chai & AI Substack, June 27) released the framework they use for every enterprise voice deployment. First gate: a 2-axis matrix — **Impact** (what scaling unlocks, what cost savings, what volume) × **Feasibility** (how mature are the backend systems, CRMs, APIs that the voice agent sits on top of). P1 = high impact + high feasibility → build now. P2 = high impact + low feasibility → fix infrastructure first, then build. P3 = low impact + high feasibility → low priority. P4 = low impact + low feasibility → rule out entirely. Case study: Amazon Ring — 100% inbound support volume, zero to production in 2 weeks. Gupta: "The fact that a process is currently staffed by humans on a phone does not make it a voice-shaped problem." Source: Chai & AI Substack (thenuancedperspective.substack.com), June 27.
**Action:** Before building any voice AI feature, run the impact × feasibility matrix. Specifically map your CRM/ticketing/data system maturity on the feasibility axis — voice fails most often not because of the voice layer, but because the systems it calls are too brittle to support it.

**CONCEPT**

**Voice AI ≠ Chat: Modality-Specific Design Rules**
From the same Vapi session: voice and chat have fundamentally different constraints that reward opposite design choices. Chat: user can reread, scroll back, click, take time. Voice: no visual reference, listener is holding concepts in working memory while listening. Vapi's production design rules for voice agents: (1) max one concept per turn — no walls of information; (2) lead with the action word — not "What I can do is help you reschedule…" but "To reschedule, I'll need…"; (3) end every turn with a closed question — give the caller one decision, not an open prompt; (4) never ask for information you already have access to — use available context first. These are the difference between a voice agent that wows in a demo and one that survives 100,000 calls. Source: Chai & AI Substack (thenuancedperspective.substack.com), June 27.
**Action:** Review any voice agent or phone-prompt you have running and apply rule (2): change every response that starts with "I" or "What I can do" to start with the verb of the action instead. This single change reduces caller drop-off in voice UX studies by 15–20%.

**CONCEPT**

**Context Windows Are Not Memory — The Core Agent Architecture Distinction**
Machine Learning Mastery editor's pick (June 26): Shittu Olumide's piece "Context Windows Are Not Memory." Core distinction: a context window is **ephemeral** — every token in it disappears when the session ends. Memory is **persistent** — survives session resets and informs future sessions. The critical error in agent architecture: treating the context window as memory leads to agents that "forget" everything on reset, require re-onboarding every session, and degrade over time as they accumulate context that will be lost. Correct architecture: external persistent memory store (CLAUDE.md, vector DB, conversation logs) + context window used only as a working scratchpad for the current task. Source: Machine Learning Mastery newsletter (jason@machinelearningmastery.com), June 26.
**Action:** Audit your Claude Code setup: does your CLAUDE.md contain everything the agent needs to bootstrap from scratch without relying on conversation history? If not, add a "Persistent Context" section that captures decisions, preferences, and project state explicitly — so the context window can always start clean.
