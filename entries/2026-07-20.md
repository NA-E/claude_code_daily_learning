**WORKFLOW**

**Claude Code Team's Thariq Shihipar: Cut the System Prompt 80% When a Better Model Ships, Use `/goal` Only With a Verifiable Finish Line**

*Source: Creator Economy / Peter Yang (peteryang+podcast@substack.com), "How I Plan, Build, and Run Loops with Claude Code," July 19</br>*

Thariq Shihipar, who works on the Claude Code team, laid out concrete workflow rules in a 40-minute interview: (1) when a stronger model ships, the first move is to remove instructions from your system prompt and CLAUDE.md, not add them — his team cut Claude Code's own system prompt by 80% because hard rules that constrained older models actively hurt newer ones; (2) `/goal` only works reliably when you can define a verifiable finish line (a rendered video, a passing test) — an open-ended goal like "build me a great game" produced weak results; (3) he plans by asking Claude to explain failure modes (e.g., how Whisper-based transcription could fail) before building, then reads the plan manually rather than trusting it blind; (4) he runs parallel Claude tasks through Slack using separate creator and verifier sub-agents, a builder/judge split this repo has now tracked multiple times in different products.

*Why it matters*: "audit your CLAUDE.md and skills every time a new model ships, and default to removing instructions rather than adding them" is a directly actionable, source-backed rule this repo hasn't captured this precisely before — most Claude Code guidance goes the other direction (add more rules).

*Who it's for*: anyone maintaining a CLAUDE.md, skills library, or system prompt across multiple Claude model versions.

*Action*: the next time Claude Code ships a model upgrade, diff your CLAUDE.md against what it looked like two model versions ago and cut anything that reads like a hard rule or example rather than a genuine constraint — that's the literal test Thariq's team applied to get an 80% reduction.

**Frontier LLMs Lose 90% of Gin Rummy Games to Casual Players — a 4.6M-Parameter Specialist Model Beats Them for $60/Year**

*Source: Every.to, "The Model Is the Easy Part" (Context Window), July 19</br>*

Every.to's Context Window column, citing work with game publisher Arkadium's Game Lab (built with Meta and DeepMind), reported that frontier LLMs — the same models making novel discoveries in math and science — lose roughly 90% of their Gin Rummy games against casual human players. Arkadium instead trained a narrow expert model with 4.6 million parameters in an 18-megabyte file that runs on a regular CPU, beats humans about 90% of the time, and costs roughly $60/year at 1 million requests/day, versus multi-millions for an equivalent-volume frontier LLM.

*Why it matters*: this is a sharply concrete illustration of a point this repo has gestured at before but never had numbers for — general-purpose frontier capability and narrow-task competence are not the same axis, and for well-defined narrow tasks, a tiny specialist model can be both better and orders of magnitude cheaper.

*Who it's for*: anyone deciding whether a task genuinely needs a frontier LLM call or could be served by a small, purpose-trained model at a fraction of the cost.

*Action*: pick one narrow, repeated task currently running through a frontier model call (a classification, a scoring, a fixed-format extraction) and estimate what a small specialist model would cost to train and run — the $60/year-vs-multi-millions gap here is the size of gap worth checking for before assuming "just call Fable/GPT" is the cheapest option.
