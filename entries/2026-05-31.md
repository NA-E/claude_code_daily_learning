**Dynamic Workflows Scope Gate — 5-Point Checklist Before Running Any Workflow (Cowork Operator)**

The first detailed best-practices guide for Dynamic Workflows dropped the morning after yesterday's launch coverage. Core argument: "Orchestration is easier to create now. Claude still needs the operator to decide whether the extra machinery belongs in the job." The author identifies the exact anti-pattern: vague requests passed to Dynamic Workflows become "a loose operating system for the task" — Claude invents scope, splits work arbitrarily, and returns "something polished enough to feel trustworthy" but potentially wrong.

Bad fits (don't use Dynamic Workflows for): short memos, one email rewrite, small spreadsheet summaries, proposal reviews where a focused single critique pass is better, pricing decisions. Good fits: codebase audits split across folders/services/packages, multi-angle research, competitive teardowns, market research divided by pain/competitor/pricing/product gaps, client prep separating call notes/account history/market context.

**Five-point Scope Gate (verbatim):** (1) Job — name the exact work. (2) Sources — list every file/folder/link/doc/repo/system Claude may inspect. (3) Split — describe how the work divides. (4) Output — define the deliverable. (5) Review — mark the human checkpoint before anything moves forward.

**Beginner-safe prompt:** *"I may want to run this as a Dynamic Workflow, but don't start yet. Help me decide whether the task deserves a workflow."* Supply: Task, Source material, Final output. Claude returns: whether the task is wide enough to split, the workstreams, source material per workstream, output per workstream, human review point, cheaper normal-Claude version, and a recommendation.

Key warning verbatim: "Messy instructions do not improve because more agents touch them."

*Source: Cowork Operator (coworkoperator.com Substack, claudedesktop@substack.com), 2026-05-31*  
**Why it matters for you:** Yesterday's entry said "check if Dynamic Workflows is accessible on your plan." This gives you the operator-level guide for using it without creating expensive errors. The scope gate checklist is the "measure twice" step before any workflow run.  
**Action:** Before launching your first Dynamic Workflow, paste the beginner-safe scope check prompt above and fill in the three fields. If Claude can't return distinct workstreams with separate source material, use normal chat instead.

**Josh Pigford's `/adversarial-code-review` Skill — Claude Opus vs. GPT Cross-Review (Behind the Craft / Peter Yang)**

After Opus writes code, Josh runs `/adversarial-code-review` — a skill that sends the output to GPT (different model family) with an adversarial review prompt: find everything wrong, be harsh. The reasoning: the same model that wrote the code is biased toward its own output. A different model family has no stake in the decision and catches different failure modes. This directly operationalizes RePPIT's "T (Test)" principle from 2026-05-30: "never use the same LLM instance that wrote the code."

*Source: Behind the Craft podcast (peteryang+podcast@substack.com), Josh Pigford interview, 2026-05-31*  
**Why it matters for you:** This is the easiest way to implement RePPIT's Test step without switching tools. You stay in Claude Code but route the review to a different model family, eliminating same-model confirmation bias.  
**Action:** Create `.claude/commands/adversarial-code-review.md` with a prompt instructing Claude to take the last code output and request a harsh adversarial review from a different LLM: "Find every bug, inefficiency, and potential failure mode. Be thorough and unsparing."

### **CONCEPT**

**LLM-as-a-Judge Bias Correction via Calibration Set (NLPiation #38)**

Raw LLM-as-a-Judge scores are systematically biased — they make weak models look better and strong models look worse than they actually are, because judges have false positives and false negatives. Prior correction methods fail by requiring knowledge of the judge's exact error rates in advance. Fix: curate a small "calibration set" of examples with known ground-truth labels, run the judge on them to measure its actual FP/FN rates, then apply a correction formula to adjust raw scores. Result: corrected accuracy matches human judgment far more closely than naive LLM-judge output.

*Source: NLPiation #38, paper: "How to Correctly Report LLM-as-a-Judge Evaluations," 2026-05-31*  
**Why it matters for you:** On 2026-05-12, you tracked Evals as "the stopping condition for prompt engineering." This paper explains why your evals might be giving you false confidence. If you're using any LLM as a judge to evaluate your prompts or outputs, you're likely reading biased scores without knowing it.  
**Action:** Before using any LLM-as-a-judge eval setup, create 20–50 examples with known ground-truth labels and run your judge on them first. Measure its error rates before trusting its scores on unknown examples.

**Attention Residuals — 1.25x Compute Efficiency via Block Attention Over Layers (NLPiation #38)**

Standard transformer residual connections sum all prior layer outputs with equal weight, causing hidden state magnitude to grow with depth and earlier-layer information to dilute. Attention Residuals replace addition with learned attention over previous layers (analogous to self-attention over tokens), letting each layer selectively weight prior layers by relevance. Efficiency fix: "Block AttnRes" applies standard residuals within layer blocks, then attention across blocks. Results: validation loss ~1.870 vs. ~1.891 baseline; matches performance of a baseline model trained with ~**1.25x more compute**. Connected to Kimi's current scaling strategy.

*Source: NLPiation #38, paper: "Attention Residuals," 2026-05-31*  
**Why it matters for you:** This is the architecture paper to watch for any fine-tuning work. The 1.25x compute gain means training 25% longer for free — significant at production scale. Block AttnRes will likely appear in major open-source model releases within 12–18 months.  
**Action:** File under "architecture patterns to check before next fine-tuning run." When evaluating model providers or open-source checkpoints, look for AttnRes or block attention mentions in release notes as a quality signal.
