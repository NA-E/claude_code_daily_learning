### **FEATURE**

**Effort Control — 4-Level Scope System; Max Should "Stay Rare" (Claude Desktop Substack)**
Anthropic added explicit effort controls: four levels mapped to task types. Low: mechanical work (rewriting emails, cleaning drafts, formatting action items). Middle: judgment work (vendor comparisons, feedback synthesis, decision memos). High: narrow audits, technical plans, migration reviews. Max: "Save for valuable work with tight scope, a reviewable output, and a clear stop point. Don't use max effort because the task is unclear. Unclear work needs scoping first." The mechanism for "more expensive": at higher effort, the model expands to cover more territory than intended, producing a swollen output you still have to clean up.
*Source: Claude Desktop Substack (claudedesktop@substack.com), "Claude Just Made Bad Prompts More Expensive"*
**Why it matters for you:** This is the underlying system behind Claude Code's `--max-effort` flag and Cowork's effort slider. Understanding the four levels turns them from buttons you ignore into scope-control tools.
**Action:** Before your next Claude task, consciously assign an effort level and state it explicitly in the prompt. "This is a medium-effort task: give me judgment, not exhaustive coverage."

**Run Classifier — 7-Point Pre-Execution Prompt; "Only Classify, Don't Execute" (Claude Desktop Substack)**
A run approval layer to place in front of any real Claude job. The 7-point classification prompt, verbatim:

```
Before doing this task, classify the run.

Task:
[Paste the task here]

Return only:

1. Recommended effort level
2. Why that effort level fits
3. Whether the task is too broad
4. The smallest useful version of the task
5. The exact output you would produce
6. The main risk if this runs too broadly
7. Where you should stop and ask me before continuing

Do not perform the task yet.
Only classify the run.
```

Separates planning from execution. No technical skill required — "better task boundaries are enough." The newsletter also gives the canonical before/after rewrite: replace `"Review this project and tell me what's wrong"` with explicit scope, file boundaries, a ranked output format, and an explicit stop condition.
*Source: Claude Desktop Substack (claudedesktop@substack.com), "Claude Just Made Bad Prompts More Expensive"*
**Why it matters for you:** This is the missing front-of-pipeline step for the RePPIT framework (2026-05-30). RePPIT starts with a Research prompt; Run Classifier starts before that — it validates the task itself before any Claude work begins.
**Action:** Copy the 7-point prompt above. Run it before your next Claude Code session instead of jumping straight into the task.

### **MODEL**

**Codex Goals (`/goal`) — 6-Part Framework; 5h45m Autonomous Run; 3,900 Emails Cleaned to 68 (How I AI / Lenny's Podcast)**
Claire Vo's solo episode on Codex `/goal`. Core distinction: "A prompt tells Claude what to do. A Goal describes what a good outcome looks like and how to verify it." Goals run an autonomous loop — work, verify, check, continue — until the defined outcome is hit. Claire's longest successful run: 5 hours and 45 minutes. The 6-part framework: (1) outcome — what should be true when done; (2) verification — how to test it; (3) constraints — what can't regress; (4) boundaries — tools and files in scope; (5) iteration policy — how to decide what to try next; (6) stopping conditions — when to ask for help. Real example: email cleanup Goal cleaned 3,900 emails down to 68 in under 4 hours, used approximately 6 million tokens. Second example: Sentry error elimination — categorize each issue, fix it, replay all historical examples; result was zero errors remaining.
*Source: How I AI / Lenny's Podcast (lenny+how-i-ai@substack.com), "Codex Goals explained & Claude Opus 4.8 review"*
**Why it matters for you:** On 2026-05-16, `/goal` shipped as Claude Code's unsupervised loop primitive. This is the first concrete 6-part framework for writing Goals that actually run for hours instead of stalling. The email cleanup example uses ~6M tokens over 4 hours — know your budget before launching.
**Action:** Write one Goal using the 6-part framework above for a repetitive task you currently do manually. Start with a narrow, verifiable outcome (email cleanup or Sentry errors are good models) and set an explicit stopping condition.

**Claude Opus 4.8 Honest Review — Greenfield Strong, Hallucination Regression, Opus 4.7 Better for Strategy (How I AI / Lenny's Podcast)**
Claire Vo tested Opus 4.8 across coding, design, and strategy in Claude Code and Cowork. New platform features shipping with it: dynamic workflows (spin off hundreds of parallel subagents) and effort controls (low to max in Claude.ai and Cowork). **Strengths:** "Easy to read, doesn't have slop tells, token-efficient, feels conversational without being annoying." Strong writing quality, improved design aesthetics (no more italicised emphasis words), good for greenfield prototypes and one-shot features. **Weaknesses — significant:** "Hallucinates when it gets stuck" — described as "a significant regression." In debugging sessions, Opus 4.8 "would make up explanations based on hypotheses rather than actual data." Also "struggles to orient itself in existing codebases," shipping edge-case bugs across cycles. **Head-to-head vs Opus 4.7 on business strategy:** Given three months of business context, Opus 4.7 "delivered numbers-anchored, structured analysis rooted in real data." Opus 4.8 "was hand-wavy, over-rotated on small data points." **Verdict:** Use Opus 4.8 for greenfield and design; use Opus 4.7 for strategy and data-heavy analysis; "test carefully for production codebases."
*Source: How I AI / Lenny's Podcast (lenny+how-i-ai@substack.com) + Skool AI Automation Society (noreply@skool.com), Nate Herk post*
**Why it matters for you:** On 2026-05-30, Opus 4.8 was described as fixing the "lazy deferral" failure mode. This review adds the other side: it introduced a hallucination-when-stuck regression. These aren't contradictory — it's more proactive on easy subtasks but more likely to fabricate explanations on hard ones. Use model selection based on task type, not one model for everything.
**Action:** For your next strategy or data analysis task in Claude Code, explicitly switch to Opus 4.7. For a greenfield prototype, use 4.8. Compare the outputs and note the quality difference in a session log.
