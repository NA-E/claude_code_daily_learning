**CONCEPT**

**"Loop Engineering, Part 1": The Five Moves a Working Loop Actually Makes**
*Source: AI Agents Simplified, issue 59 (aiagentssimplified@substack.com)*

Google Chrome engineer Addy Osmani coined "loop engineering" in a June 2026 essay: replacing yourself as the person who prompts the agent by building a system that prompts it for you. The newsletter reports that Peter Steinberger (creator of OpenClaw) and Boris Cherny (creator of Claude Code, now leads it at Anthropic) independently said near-identical things the same week — Cherny stating he no longer prompts Claude directly, "loops do that and figure out what needs doing." The piece defines a 4-layer stack (prompt engineering → context engineering → harness engineering → loop engineering) and names the five moves every working loop must make: Discovery (finds its own work, e.g. reading CI failures), Handoff (hands the task to an agent in an isolated workspace, e.g. a git worktree), Verification (a *different* agent checks the work), Persistence (writes down what happened somewhere that survives the conversation), and Scheduling (runs again tomorrow without a human pressing a button).

*Why it matters*: this repo's own June 26 entry flagged "Loop Engineering named by Claude Code's creator" as a concept to watch — this is the first full framework for it, and this exact digest routine is a working example of four of the five moves (Discovery, Handoff via git commit, Persistence via `entries/`, Scheduling) but is missing the fifth.

*Who it's for*: anyone building a recurring Claude Code/agent routine — including the one that wrote this digest.

*Action*: audit any recurring automation (this digest routine included) against the five moves; name which one is missing before building anything new.

**CONCEPT**

**The Self-Grading Trap: Why an Agent Can't Judge Its Own Work, and What Anthropic Does Instead**
*Source: AI Agents Simplified, issue 59*

Part 2 of the same piece: Anthropic engineer Prithvi Rajasekaran documented in an internal engineering write-up that agents asked to evaluate their own output tend to confidently praise it, even when a human can immediately see it's mediocre — because the agent is still inside the reasoning that produced the result. His fix, borrowed from GAN architecture: split builder and judge into two separate agents with different instructions, and for frontend work, have the judge open the actual page with browser automation, click through it, and screenshot it rather than just reading the code. Claude Code has turned this into a shipped command, `/goal` — you give it a condition ("all tests in test/auth pass and lint is clean") and a fresh, separate model checks after every turn whether that's genuinely true.

*Why it matters*: this is a direct, sourced answer to the open question this repo's own July 16 morning entry raised — whether repeatedly flagging the Knwn-Local-App cron without a verification/enforcement step is itself the failure mode. This routine currently self-grades ("13 entities stored" is asserted, never independently checked).

*Who it's for*: anyone running an unattended Claude Code loop that reports its own success.

*Action*: try `/goal` in Claude Code on one real task this week (e.g., "all tests pass and lint is clean") and compare its verdict against your own read of the result — then consider whether this digest routine needs a second, independent pass that checks entity accuracy before the file is committed.

**CONCEPT**

**Every.to's "The Case Against Skills": When a Skill Helps vs. When It Fights the Model**
*Source: Every.to, "Context Window" (hello@every.to), by Laura Entis*

Mike Taylor, Every's head of tech consulting, argues most trending Claude Skills libraries are unnecessary bloat: frontier models like Fable 5 and GPT-5.6 have already absorbed the reasoning most skills try to hand-hold, so extra instructions can fight the model's own training and increase mistakes and cost rather than reduce them. His rule: a skill earns its place only with proof it improves outcomes, and is genuinely useful mainly when you need an exact, specific output format — his example is a custom PowerPoint template built to a brand style guide, or a workflow referencing internal company data the model can't otherwise know.

*Why it matters*: directly cuts against the "sprawling skill library as status symbol" trend on X that the piece describes, and complicates this repo's own July 16 morning framing of "Claude Code skill as a deliverable" (the Google Ads Builder skill) — the two pieces landed the same day with opposite takes on skill proliferation.
