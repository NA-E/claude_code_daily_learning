**Claude Kit v4: Free, Open-Source Risk-Scored Quality Gate for Claude Code and Codex**

*Source: Circle.so community notification (no-reply@notification.circle.so), "Claude Kit - Free / Open Source v4 released," posted by Brett Stark in WINS, July 22, 2:16 AM UTC*

Brett Stark released claude-kit under MIT license after building it as internal infrastructure over ~10 months, then open-sourcing it once Claude/GPT capability caught up to what it replaced. Install with `/plugin install bs@buildproven` — no symlinks or config wrangling. It ships: a risk-scored quality gate (`risk-score.js` reads your diff and scales review depth — a typo gets a light pass, a CI workflow change auto-escalates to max scrutiny); a run governor that caps runaway review loops and fails closed rather than hanging or silently passing; 17 `/bs:*`, `/gh:*`, `/cc:*` commands; 35 skills (including an autonomous "ralph" workflow and multi-LLM strategy panels); 14 specialist agents (security auditor, code reviewer, accessibility tester, performance engineer); and 415 tests at 85%+ line coverage on the gate itself. It's provider-neutral (falls back between Claude Code and Codex on quota/auth/timeout) and deliberately drops features Anthropic already ships natively (background agents, worktrees, memory, `/loop`, `/code-review ultra`) to avoid duplicating the platform.

*Why it matters*: this is a direct, installable answer to the exact gap this repo has tracked since July 16 — a risk-scored gate plus a fail-closed governor is close to the creator/verifier split Thariq Shihipar described on July 20, except here it's a maintained open-source package with its own test suite instead of something to hand-build from a two-prompt session.

*Who it's for*: anyone running Claude Code or Codex on a real codebase who wants review depth to scale with actual diff risk instead of one-size-fits-all review.

*Action*: run `/plugin install bs@buildproven` on a low-stakes branch of this repo (or another active project) and compare its risk-scored gate against the July 20 Top Thing's hand-built verifier plan before deciding which approach to commit to.

**Loop Engineering in Production: Stripe's "Minions" Land ~1,300 PRs/Week; Addy Osmani Runs a Personal Loop Before Breakfast**

*Source: AI Agents Simplified, issue 60 (aiagentssimplified@substack.com), "What You Should Know About Loop Engineering (Part 2)," July 21, 7:45 PM UTC*

Two concrete, named examples of loop engineering at opposite scales. Stripe built an internal system nicknamed "Minions" — per engineer Steve Kaliski on the *How I AI* podcast, you tag the bot in Slack or react with an emoji, and Stripe's agents were landing around 1,300 pull requests a week with no human writing the code, only reviewing it. Before the agent starts "thinking," a separate hard-coded system gathers the relevant tickets, docs, and code so nothing predictable and rule-based gets handed to the probabilistic model. At the other end, Addy Osmani runs a personal loop every morning that reads yesterday's CI failures, open issues, and recent commits on its own, opens an isolated workspace per fixable item, has one agent draft a fix and a second review it, opens a PR automatically when it passes, and drops anything uncertain into an inbox for him — nothing requires him to be awake, and nothing merges without a check.

*Why it matters*: both examples separate the creator agent from the reviewer agent — the exact split Thariq Shihipar described on July 20 and this repo asked about building on July 21 — now with a real weekly-PR number (1,300) attached to the enterprise version and a solo-developer version small enough to copy directly.

*Who it's for*: anyone maintaining a codebase with recurring, well-scoped fix work (CI failures, small PRs) who reviews code more than they write it.

*Action*: if you want the Osmani version specifically, the shape is: agent reads yesterday's CI failures + open issues → isolated workspace per item → draft agent + separate review agent → auto-PR on pass, inbox on uncertain — copyable as a weekend project, not a platform purchase.

**The Four Costs of a Loop Going Wrong: Verification Debt and Comprehension Rot Don't Show Up Until They Compound**

*Source: AI Agents Simplified, issue 60, July 21, 7:45 PM UTC*

The same Loop Engineering Part 2 issue names the failure modes that don't appear while a loop is running well. "Verification debt": every PR a loop opens without a real check is a small loan against future time — it looks fine because the tests that exist are green, and the problem lives in what the tests don't cover, compounding quietly until it surfaces all at once. "Comprehension rot": the faster a loop ships code you didn't write, the bigger the gap between what your codebase actually is and what you think it is.

*Why it matters*: this reads as a direct risk checklist for this routine's own construction — a Discovery/Handoff/Persistence/Scheduling loop with no independent Verification step (the exact gap named July 16) is precisely the shape this piece warns accumulates verification debt silently.

*Who it's for*: anyone running an unattended agent loop against a real codebase or a real publishing pipeline (including this routine).

*Action*: before adding more automation to any existing loop, ask specifically what fraction of its output has a real check versus a green-but-untested pass — that fraction is the verification debt number this piece says compounds invisibly.

**RadLE 2.0: A Radiology Benchmark That Scores Whether AI Knows When to Say "I'm Not Sure" — Plus the 1-3-1 Prompt Format**

*Source: The AI Exchange, edition 194 (theaiexchange@mail.beehiiv.com), "Doctors just built a test for whether AI knows it's wrong," July 21, 1:33 PM UTC*

Radiologists built RadLE 2.0 (technical report at crashlab.in/radle-technicalreport), a benchmark that adds a second score beyond raw accuracy: can the AI recognize when it's out of its depth and hand the case back to a human? The newsletter's stated fix is to explicitly give a model permission to express uncertainty, and specifically to use a "1-3-1" prompt format — state the situation, give three options, commit to one pick with reasoning — because judging a pre-weighed 1-3-1 answer takes about 15 seconds versus silently redoing the model's reasoning to check a single confident answer.

*Why it matters*: "a confident wrong answer is more dangerous than an honest 'I don't know'" is a direct, quotable design principle for any prompt or CLAUDE.md instruction set that currently rewards a model for always producing a clean, decisive answer regardless of certainty.

*Who it's for*: anyone writing system prompts, CLAUDE.md files, or agent instructions where a wrong-but-confident answer is costlier than an honest "not sure."

*Action*: add one line to a prompt or CLAUDE.md you use regularly: "Before your answer, give three options, tell me which you'd pick and why, then list anything you weren't sure about or had to assume" — and time how much faster you can check it versus a single-answer response.

**Whoop's Prototyping Binge: Fast AI Prototypes Didn't Make Product Decisions Easier — They Made Them Harder Without a Filter**

*Source: Every.to (hello@every.to), "Drowning in Demos? Here's a Better Way to Prototype," by Hilary Gridley (former Whoop product lead), July 21, 4:46 PM UTC*

Gridley describes leading Whoop's product team through an AI-prototyping binge once tools like Bolt and Replit made it possible to spin up a working prototype in an afternoon. The problem: velocity went up, but there was no consistent process for what happened after a prototype existed — occasionally one generated enough excitement that a team would drop its prior plan and build it, with no structured way to judge which ideas were actually worth pursuing.

*Why it matters*: this is a direct, named counter-example to "more prototypes, faster, is unambiguously good" — velocity without a decision filter just moves the bottleneck from building to deciding, which is a pattern worth checking against any team that has adopted fast AI prototyping tools without also building a triage step.

*Who it's for*: any product team that has adopted AI prototyping tools (Bolt, Replit, Claude, or similar) and noticed decision paralysis rather than faster shipping.

*Action*: if your team prototypes fast now, add one explicit checkpoint — a required question like "what specific decision does this prototype answer, and who signs off before it goes further" — before building the next one.
