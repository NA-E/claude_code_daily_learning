**LiteLLM AI Gateway — Self-Hosted Reverse Proxy for All LLM Traffic (The Neural Maze Issue #3)**
The Neural Maze published a production systems-engineering deep dive on why every serious agent deployment ends up behind a reverse proxy — and how to build one with LiteLLM in 5 minutes. The core insight: AI teams are repeating the 2010 web architecture mistake. Every app calls OpenAI directly with the raw key, retry logic is reinvented from scratch in each repo, and when the key leaks someone loses an afternoon rotating it across 17 places. The fix is the same fix as nginx in 2010: put a box in the middle. LiteLLM is that box: 140+ providers, 240M+ Docker pulls, single YAML config. Three patterns the docs bury: (1) **Key federation to escape rate limits** — register the same model twice under different API keys, router round-robins, effective RPM doubles (e.g., 500 RPM × 2 keys = 1,000 RPM with automatic circuit-breaker); (2) **Two hidden fallback types** — `content_policy_fallbacks` (if OpenAI refuses a prompt, transparently retry on Claude) and `context_window_fallbacks` (if input is too long for gpt-4o-mini, auto-escalate to gpt-4o); (3) **Cost-based routing** — name one logical endpoint `cheap-tier` and back it with Gemini 2.0 Flash + GPT-4o-mini + Claude Haiku-4-5, `routing_strategy: cost-based-routing` in YAML, router picks cheapest available per call. The 2026 plot twist: LiteLLM now proxies MCP servers too — one `mcp_servers:` block in config.yaml gives every app behind the gateway access to GitHub MCP, Brave Search, etc., with zero MCP wiring on the client side. Exact YAML configs provided in the issue. Self-host: `docker run -p 4000:4000 -v $(pwd)/config.yaml:/app/config.yaml ghcr.io/berriai/litellm:main-latest`. Full issue: https://theneuralmaze.substack.com/p/dont-marry-your-llm-provider
Source: The Neural Maze Issue #3 (theneuralmaze@substack.com, May 19).
**Why it matters for you:** The Routines + /goal architecture (May 16) and the agent skill stack (May 12) both make direct API calls to Claude. If you ever add a second model (say, Cursor Composer 2.5 at $1/task for classification tasks), you'll want a gateway rather than scattered API keys. The MCP proxy capability is particularly relevant — one gateway config replaces individual MCP server wiring per project.
**Action:** Read the full Neural Maze issue. Before anything else, run the minimal 3-provider LiteLLM config locally (`docker run` command above). In 5 minutes you'll have a local proxy at `localhost:4000` that you can point Claude Code at instead of calling Anthropic directly — and you can observe cost per call in real time.

### **CONCEPT**

**"Don't Marry Your LLM Provider" — AI Gateway as the Production Agent's Reverse Proxy (Neural Maze)**
The Neural Maze Issue #3's central argument: teams building agents are making the same mistake web teams made in 2010 — every app calls the API directly with the raw key. The fix is architectural, not code: a reverse proxy (AI gateway) that sits in front of all model calls. Quoting directly: *"The gateway is becoming the single chokepoint for all agent runtime traffic — model calls, tool calls, sub-agent calls."* The critical distinction the issue draws: the proxy server polices the door (auth, rate limits) while the Router does actual routing — these are separate components in LiteLLM. The "life of a request" sequence: auth (virtual key, cache-first then DB) → rate limits (4 parallel checks, refuses before any provider call) → routing (provider selection, fallbacks) → provider call → response. Everything observability-related runs asynchronously after the response, which is why a well-configured LiteLLM gateway adds only "single-digit-millisecond latency to a model call." The asymmetry between sync hot path and async observability is what makes it feasible in production. The practical framing: *"Your model is rented. Not 'I could swap providers if I wanted to.' 'My gateway is constantly swapping providers and my code never sees it.'"*
Source: The Neural Maze Issue #3 (theneuralmaze@substack.com, May 19).
**Why it matters for you:** Every MCP server integration in your current stack (Weavable from May 16, any others) is a potential point of vendor or credential lock-in. The LiteLLM gateway's MCP proxy capability means you can centralize all of that into one config file — and rotate credentials, add providers, or change routing logic without touching any application code.
**Action:** Read the full issue and internalize the "three patterns the docs bury" section. Start with Pattern C (cost-based routing) since it has the clearest immediate ROI: one YAML change routes cheaper model calls automatically as pricing changes across providers.
