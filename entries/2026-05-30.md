### **WORKFLOW**

**RePPIT Framework — 5-Step Production-Ready AI Coding (Mihail Eric, Chai & AI Substack)**  
Developed by Mihail Eric (Head of AI at Monaco, Stanford lecturer), RePPIT is a structured 5-step methodology claiming 2–3x faster production-quality code delivery over "vibe coding." Each step is codified as a reusable prompt:  
- **R (Research):** Prompt Claude to read your codebase and output a 100–300 line Markdown research doc covering design decisions, file layout, tech stack, dependencies — with zero mention of the feature being built. Do not persist this doc in the repo; regenerate it fresh each time to avoid staleness. External library docs can be pulled live via Context7 MCP.  
- **P (Propose):** Input: research doc + feature description → output: exactly 2 distinct proposals covering overview, trade-offs, validation strategy, open questions. "Two is the sweet spot" — three or four produces repetitive minor variations.  
- **P (Plan):** Populate a design doc template including functional/non-functional requirements, major design decisions with rationale, implementation phases, explicit out-of-scope list. Key insight: "reviewing a planning document is easier than reviewing a pull request" — share with teammates before writing code.  
- **I (Implement):** Execute against the design doc. With a detailed spec, "the model has very little room to guess or wander off."  
- **T (Test):** Independent review — **never use the same LLM instance that wrote the code.** Use a different LLM family entirely, or at minimum clear the context completely before reviewing. Demo used: SQL ILIKE vs. PostgreSQL TSVector full-text search proposals on a Medium clone.  
*Source: The Chai & AI Community (thenuancedperspective Substack), Mihail Eric, 2026-05-30*  
**Why it matters for you:** You've been tracking Claude Code production workflows since 2026-05-10. RePPIT closes the gap between the CLAUDE.md context system (2026-05-19) and the Maven cohort (starting June 2) — it's the methodology that makes both investments pay off.  
**Action:** For your next Claude Code feature, run Step R first: prompt Claude to read your codebase and output a 100–300 line research doc with no mention of what you're building. This single step eliminates the "explain my project again" tax from every session.

**Compound Engineering v2 — 8-Step Loop (Kieran Klaassen, every.to Guides)**  
Kieran Klaassen updated the original 4-step compound engineering loop (Brainstorm → Work → Review → Compound) to 8 steps: **Ideate → Brainstorm → Plan → Work → Review → Polish → Compound → Repeat.** Two new front steps: Ideate (decide what's worth building: user, product, edge cases, what feels exciting) and Brainstorm (exploration before planning). One new end step: Polish (the feel check — design, copy, experience — not just "does it technically pass"). Core philosophy: "The middle of a lot of work will get automated. If you want the work to be good and feel like yours, you still need to be there at the beginning and the end." Humans are the bread; AI is the filling. Open-source Claude Code plugin: github.com/everyinc/compound-engineering-plugin. Compound Engineering Camp 3 with Trevin Chow runs June 5.  
*Source: every.to Guides (hello@every.to), Kieran Klaassen, 2026-05-29*  
**Why it matters for you:** This maps directly to your daily digest pipeline. The "Ideate" step (what's worth capturing today?) and "Polish" step (does this entry actually read well?) are currently absent from this autonomous run — adding them would improve output quality at zero extra cost.  
**Action:** Read the updated guide at every.to/guides/compound-engineering. Apply Ideate (5 min before starting) and Polish (5 min reviewing output) to your next Claude Code project.
