**Boris Cherny's 10-Step Claude Code Workflow — Auto Mode, Worktrees, Memory-to-CLAUDE.md (The Neuron)**
Boris Cherny (creator of Claude Code) gave a casual overview of his daily workflow, documented by The Neuron. The 10 steps: (1) Use Claude Code desktop app — handles worktrees (separate repo copies for parallel agents) automatically. (2) Open **agent view** in terminal alongside desktop — replaces "six terminal tabs" with one dashboard. (3) One scoped task per agent — give each a specific job, let it run as its own session. (4) **Use auto mode for everything** — Boris says newer models need less planning, so start in auto-mode and move to the next task. "Safer because it only asks permissions for the most important stuff." (5) Turn repeated mistakes into CLAUDE.md memory — when Claude makes the same mistake twice, update CLAUDE.md or create a reusable skill. (6) **Have Claude verify by running the thing, not running tests** — Boris says make the final check behavioral ("does it actually work for the user?"), not test-driven. Research note: Kun Chen found TDD makes agents overfit to weak tests and stop early, with worse pass rates and higher token use in ProgramBench evaluation. (7) Move recurring work into `/loop` or `/goal` (goal = loop with a defined completion condition). (8) Use **Remote Control** (`/remote-control` command) to monitor and start agents from phone. (9) Use voice mode for ideas on the fly. (10) **Keep context minimal** — goal, constraints, and a pointer to where Claude can find more. No micromanagement.
*Source: The Neuron (theneuron@newsletter.theneurondaily.com), June 14, 2026*
**Why it matters for you:** This repo has been building toward a Boris-style setup since 2026-05-10 (Claude Code OS architecture), 2026-05-12 (Routines + Dreaming), 2026-05-16 (`/goal`), 2026-06-08 (adversarial multi-agent patterns), 2026-06-11 (sub-agent YAML). **Braintrust Evals-as-PRD — Feedback Loop Architecture, Human Taste Amplification, Carving vs Constructing (How I AI / Lenny's)**
Ankur Goyal (Braintrust CEO) on the highest-leverage AI engineering decisions: (1) **Evals are the modern PRD** — they define success criteria (what does "good" look like, quantifiably), not implementation steps. The best evals include concrete test cases and scoring functions. (2) **Build the eval → real-world data pipeline first** — for AI product teams, the #1 engineering priority is "a pipeline that summons real-world data and converts it into evals." This is the CI equivalent for AI products. (3) **Run 4–6 foreground agents simultaneously** — that's the human concurrency limit for managing context across parallel sessions. (4) **When an agent fails: close the session, improve the evals, start fresh.** Don't prompt your way out of failures. Ankur spent a weekend hand-writing a 3,000-line eval after vibe coding turned it into "trash." (5) **Product building is carving, not constructing.** When customers complain, Braintrust removes the thing 90% of the time. Remove complexity first. (6) **Quantify expert taste so it scales.** Ankur runs hundreds of evals, then asks his designer David for a vibe check every few days. David's feedback ("it's OK to show both languages as long as...") gets encoded into scoring functions. David's quality bar applies to more decisions than David can ever review.
*Source: How I AI / Lenny's Newsletter (lenny+how-i-ai@substack.com), June 15, 2026*
**Why it matters for you:** The 2026-05-12 entry first captured "Evals give prompt engineering a stopping condition." The June 11 LangChain ADLC entry confirmed traces + evals as the CI equivalent for agents. Today's Braintrust interview is the production practitioner's version: how a real team runs it at scale. The carving metaphor is especially useful for this repo — the instinct has been to add entities and skills. The Braintrust insight says removing is the higher-leverage move.
**Action:** For your most-used Claude Code skill, write one eval: what does a correct output look like, quantifiably? The minimum viable eval is: 3 sample inputs + expected outputs + a 1-line pass/fail rubric. This is the eval pipeline's Day 1. Once you have one eval, adding more is fast.

**Headroom — 60–95% Token Compression, Open Source, Applicable to Agent Tool Outputs (The Neuron)**
Headroom (`github.com/chopratejas/headroom`; open source, free) compresses tool outputs, logs, files, and retrieval chunks **before they hit the model context window** — achieving 60–95% token reduction while preserving answer quality. Works by identifying semantically redundant content and collapsing it. Direct use case: agent tool call results that return large payloads (API responses, file reads, search results) before they fill context.
