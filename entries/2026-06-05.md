### **WORKFLOW**

**Agentic NPC Architecture — Context Discipline, State Validation, Rejection Feedback, Dual-Call Planning (AI Agents Simplified)**
A post-mortem from building "Unwritten Realms," a multi-NPC text game where every character is an LLM agent. Core finding: "A believable agent is almost never a model problem. It's a context problem and a grounding problem." All characters share the same base model; what differentiates them is entirely the assembled context. The architecture has four components:

- **Context discipline:** 6-field character object (persona, float-intensity traits, private facts, goals, rolling memory summary, dialogue). Assembled fresh every turn. Selection is the craft: "Dump everything and you blow the token budget and bury the handful of facts that matter."
- **State validation:** "The model never touches game state. It proposes." Every action is a structured tool call. An external validator sits between proposal and world — `giver.inventory.get_item(self.item_id)` etc. State lives in the engine; an action either passes or is rejected and the world is unchanged.
- **Rejection as signal:** Initial architecture failed-fast (error logged, turn moved on). Problem: character narrated as if action succeeded, causing state drift. Current architecture returns rejection as natural language: `"you tried to give the map; you don't have it"` — model revises within same turn. "Grounding isn't a one-shot filter. It's a short conversation between the model and the engine, and the engine always gets the last word."
- **Dual-call planning:** Two model calls per turn — Call 1 (cheap planner): structured tactical intention, e.g., `"stall; find out why they want the map before giving anything up"`. Call 2 (performance): executes the intention in character. "The strategy is the model's private reasoning; the dialogue is the performance. The player never hears the gears turn."

*Source: AI Agents Simplified (aiagentssimplified@substack.com), Amirali Lotfi, June 4, 2026*
**Why it matters for you:** On 2026-05-11, CooperBench showed multi-agent coordination is 30% less reliable than single-agent. This article is the first concrete framework for *why* that happens and *exactly how to fix it*. State drift from unvalidated proposals is the mechanism; external validation + rejection-as-signal is the fix. This directly applies to any Claude Code workflow that touches external state (databases, APIs, file systems).
**Action:** In your next Claude Code agentic session, implement the rejection-as-signal pattern: when a tool call fails, return the failure reason as natural language to the model rather than logging silently or raising an exception. This single change prevents state drift and allows the model to self-correct within the same turn.

### **CONCEPT**

**Float-Valued Trait Intensities — Behavior Control Without Fine-Tuning (`greedy 0.8` vs `greedy 0.3`)**
From the AI Agents Simplified article: character behavior is controlled via float-valued trait fields, e.g., `cautious 0.7`, `greedy 0.4`. These appear in the system context assembly. Findings: `greedy 0.8` reliably makes a character bargain harder before parting with an item; `greedy 0.3` makes the same character give things away freely — with zero fine-tuning required. Traits-with-intensities act as "cheap, interpretable control knobs" because they are parsed directly by the model's language understanding of the word plus the float's magnitude.
*Source: AI Agents Simplified (aiagentssimplified@substack.com), June 4, 2026*
**Why it matters for you:** Any agent you build with Claude Code that has variable behavior (urgency, formality, risk tolerance, creativity) can be controlled by float-valued fields in the system prompt rather than separate model fine-tuning or complex prompt engineering. This is a simple technique with high control leverage.
**Action:** In your next Claude Code agent, add a `behavior_traits` dict to your system prompt: `{"urgency": 0.8, "formality": 0.3, "risk_tolerance": 0.5}`. Adjust the floats per task context rather than rewriting the entire prompt. Test whether this produces measurable behavioral variation.

### **NEWS / CONCEPT**

**Fine-Tuning LLMs Unearths Pretraining Text — BMC@5 >40% for 36/51 Books, Verbatim Spans up to 440 Words (Stony Brook/CMU/Columbia)**
Stony Brook University, Carnegie Mellon University, and Columbia Law School researchers showed that fine-tuning LLMs on a benign writing task (expanding plot summaries) causes models to reproduce large verbatim portions of pretraining books, bypassing copyright alignment. Models tested: DeepSeek-V3.1, Google Gemini 2.5 Pro, GPT-4o. Baseline GPT-4o produced 7.36% Book Memorization Coverage (BMC@5 = % of book reproduced in 5+ word contiguous spans). After fine-tuning on Murakami novels, tested on 51 books by 32 other authors: **BMC@5 exceeded 40% for 36 of 51 books; GPT-4o reached 91.9% BMC@5 in one case.** All three models generated verbatim spans up to **440 words.** Fine-tuning on public-domain text produced BMC@5 near 0 — confirming the effect targets pretraining data specifically.
*Source: The Batch / DeepLearning.AI (thebatch@deeplearning.ai), June 5, 2026*
**Why it matters for you:** If you fine-tune any model (via API or local) on content generation tasks and plan to deploy publicly, this is a live copyright risk — alignment training suppresses verbatim output but does NOT erase encoded strings. Fine-tuning on generation tasks teaches the model to decode them. Claude Code's base model carries the same risk if fine-tuned.
**Action:** Note this finding as a constraint: do not fine-tune Claude or any frontier model on verbatim writing tasks (ghost-writing, paraphrase expansion, style mimicry) without reviewing outputs for BMC. Use synthetic training data for generation task fine-tuning to avoid triggering pretraining recall.

Start with this 5-line scaffold in `mcp_server.py`:

```python
from mcp.server import Server
from mcp.server.stdio import stdio_server

server = Server("my-first-mcp")

@server.tool()
async def get_today_context(query: str) -> str:
    # Return anything Claude needs: project notes, live data, API results
    return f"Context for '{query}': [your custom data here]"

if __name__ == "__main__":
    stdio_server(server)
```
