### **WORKFLOW**

**Evals-Based Prompt Engineering — Write Tasks First, Loop Until They Pass**
From the "Code with Claude 2026" conference (reported by TechTiff): Evals turn prompt engineering into a process with a measurable stopping condition. The protocol: (1) Write 5–50 real inputs + acceptable output descriptions as your task set — before touching the prompt. (2) Write a preliminary prompt. (3) Run the prompt against all tasks. (4) See which tasks fail and specifically why — format drift, missed step, wrong reasoning type. (5) Fix the one change the failure points to. (6) Run again. Repeat until tasks pass. The prompt is finished when all tasks pass. Without evals, prompt engineering is wording-guessing with no stopping criterion. Source: TechTiff, citing Code with Claude 2026 conference practitioner sessions (May 12).
**Why it matters for you:** Every skill you've been building (completion receipts, brand CLAUDE.md, session context) still has the same quality problem: you can't tell if a prompt is done or just looks done. Evals add the feedback loop.
**Action:** Pick your single most-used Claude Code prompt (likely the one that kicks off your weekly planning or client research). Write 5 real inputs you've used before. For each, write a 2–3 sentence description of the output you'd accept. Run the prompt tonight. Record how many pass. That's your first baseline.

**"Fewer Agents, More Tools" Rule — Split Only When Workload Actually Demands It**
The strongest multi-agent insight from Code with Claude 2026 (TechTiff, May 12): multi-agent systems fail when responsibility gets muddy. The three rules that held up across every successful multi-agent setup: (1) Keep context windows clean. (2) Sandbox agents. (3) Separate understanding agents from writing agents — reading source material in one context window, producing output in another. When a system misbehaves, the fix is almost always fewer agents and better tools, not more agents. Split out a new agent only when the workload actually demands it. This directly extends the CooperBench finding from May 11 (30% lower success rate in multi-agent vs solo) with practitioner validation from a production conference. Source: TechTiff, Code with Claude 2026 (May 12).
**Why it matters for you:** The temptation when building Claude workflows is to add orchestration complexity early. This rule gives you a clear heuristic: stay single-agent until the workload forces a split, then separate understanding from writing before separating anything else.
**Action:** Audit your current Claude Code workflows. For any workflow where you're using more than one agent or sub-agent: map which agent is "understanding" (reading/analyzing) and which is "writing" (producing output). If they're mixed in one context window, that's your first separation to make — before adding any new agent.
