### **CONCEPT**

**Eval Deployment Three-Tier System (Hamel Husain, Flashcard #12)**
Hamel Husain (Parlance Labs) published the deployment companion to the evals series. Three contexts, each with different goals: **CI/CD** — runs before merge, 100+ curated examples covering core features + past bugs + edge cases, favor code-based assertions over LLM judges, blocks merge on failure. **Online Monitoring** — async after response sent, can use LLM-as-Judge since not blocking user requests, track metrics with confidence intervals, surface new failure patterns that feed back into CI. **Guardrails** — synchronous in request path, must be fast with low false-positive rates, stick to code-based checks or small classifiers (LLM judges too slow and unreliable here). These form a continuous improvement loop. Visual flashcard: https://hamel.dev/notes/llm/evals/flashcards/12-deploy-evals.png
*Source: Hamel Husain / Parlance Labs, 2026-05-26*
**Action:** Draw this three-tier map for any AI system you're building or planning to build. For each tier, answer: (1) what dataset do you run, (2) what evaluator do you use, (3) what's the threshold that triggers action.

## Top Thing to Implement Right Now

**ONE ACTION → HIGH LEVERAGE → DO TODAY**

**Wire up the three-tier eval deployment system for any AI feature you're currently building.**

From Hamel Husain's Flashcard #12 (Parlance Labs, 2026-05-26):

1. **CI/CD tier:** Create a folder `evals/ci/` with 100+ test examples (your known failure modes + core feature cases). Write one code-based assertion (not LLM judge) that runs on every PR. This is the minimum: `pytest evals/ci/test_core.py`.
2. **Online Monitoring tier:** Add one log line per AI response that writes to a file or DB with: timestamp, input hash, response, and a flag field. Set a cron job that samples 10% of live traffic and runs an async LLM-as-Judge check. Threshold alert when quality drops below baseline.
3. **Guardrails tier:** Identify one high-impact failure mode (e.g., hallucinated citations, PII leakage, off-topic responses). Write a synchronous code-based check for it that runs before the response hits the user. Keep it under 50ms.

This connects directly to the May 12 entry (building evaluators) — today's flashcard is the natural next step: you have evaluators, now you need to deploy them in the right contexts.
